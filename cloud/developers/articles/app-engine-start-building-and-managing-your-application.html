Exercise 1. Start Building and Managing your Application

In the previous exercise, you deployed and tested the completed Conference Central application, to get an idea of what it does. It is now your turn to build the Conference Central application, almost (but not quite) from scratch.

To save you from spending too much time creating skeletal HTML pages, we have created a “stub” version of the application to get you started.

In this exercise, you will download a zip containing the skeletal application, explore the layout of the application, then deploy the application and investigate it in the Admin Console.

What You Will Learn

You will learn how to configure an application, how to deploy an application to the local development server and also to production, and how to find your way around the Admin Console.

What You Will Do

In this exercise, you will:

  1. Download a skeleton application.
  2. Tour the pages to learn the organization of the application.
  3. Configure the application.
  4. Deploy and run the application (even though it does not do much yet).

Get Ready

1. Download a zip containing the skeletal application.

  • Download and install the zip containing the initial version of ConferenceApp_1.
    • Java: ConferenceApp_1_Java_stub
    • Python zip: ConferenceApp_1_Python_stub

    For instructions, see Downloading and Installing a Zip File for an Application in Tasks for Working with App Engine Applications.

    Java note: If you have updated the Java App Engine SDK, see xx for information about how to update the project in Eclipse to use Updating Projects to Use the Latest App Engine SDK in in Tasks for Working with App Engine Applications.

2. Rename ConferenceApp_1_START to ConferenceApp.

Understanding the Organization of the Application

After downloading and installing the zip file containing the skeletal application, you will now explore the source code.

[ Go to Python steps ]

Java -- Explore the Application

The Java version of the application uses both servlets and JSPs. You will use Eclipse to explore the code.

JSPs are servlets defined using static content (such as HTML) mixed with Java code. JSP lets you put Java inside HTML pages. Using JSP, a page can switch back and forth between HTML and Java. JSP is not specific to App Engine, and this tutorial does not attempt to teach it or explain its syntax. There are many books and websites that teach JSP. However, just to get you started, we provide a JSP Quick Reference that summarizes the syntax that is used in this tutorial.

You can read more about using JSPs with App Engine in the developer documentation:

  • Using JSPs
  • Configuring JSPs

1. In Eclipse, explore the contents of ConferenceApp.

  • In the src folder you see the SaveUserProfile servlet under com.google.cloud.training.conference:
    The project also contains a folder more_classes that contains additional servlet classes that you will move into the main war file as needed as you work through the exercises.
  • In the war folder, you see several JSPs files.

2. Take a look at header.jsp.

This file defines the heading that appears on every page of the Conference Central application. It writes the <head> section, links in the stylesheet, starts the <html> and <body> sections, displays an image for the banner, and then writes the navigation links under the banner image.

You see several lines marked “TODO.” As you work through the exercises, you will fill in the TODO sections.

Take a look at one of the navigation links:

<span class="nav-item"><a href="/listconferences">Upcoming Conferences</a></span>

If you were to deploy the application now and try it out, the link above would not work, because the URL /listconferences is not defined yet. You will define it soon. For now, continue inspecting the other JSP files in war.

3. Take a look at some of the other JSP files.

The war directory includes several other JSP files, which do the following:

  • include the header
  • define an <H1> heading
  • include partially completed content
  • include the footer

As you work through the exercises in this training, you will fill out each of these pages so that the application will be functional.

The following JSP files define content that is common across multiple JSPs files:

  • footer.jsp -- the footer for every page
  • header.jsp -- the header for every page
  • constants.jsp -- contains constants, such as lists of cities

The following JSP files correspond to the pages that are linked from the home page and the horizontal navigation bar:

  • homepage.jsp -- the home page for the application. Currently it just links to other pages.
  • listconferences.jsp -- this page will eventually list the conferences that have been scheduled.
  • scheduleconferences.jsp -- this page displays a form for scheduling a new conference.
  • userprofile.jsp -- this page displays the user’s profile, including the topics that they are interested in learning about.
  • venues.jsp -- this page will eventually display information about the venues for conferences.
  • developer.jsp -- a page for developers (which is only accessible if you are logged in as a developer/admin)

The project also contains a folder more_JSPs that contains additional JSPS that you will move into the main war file as needed as you work through the exercises.

Python -- Explore the Application

The Python version of the application uses the Jinja2 templating language. Jinja2 is modelled after Django’s templates. Jinja2 is not specific to App Engine, and this tutorial does not attempt to teach Jinja2 or explain its syntax. For information, see the Jinja2 website.

1. In the ConferenceApp directory where you unzipped the downloaded zip file, explore the contents of the application.

You see a python module:

  • main.py

and several jinja2 template files. Note that the .template extension is arbitrary. Jinja2 does not impose any naming convention. You also see app.yaml and some other folders and files.

2. First take a look at base.template. This file defines the heading that appears on every page of the Conference Central application. It writes the <head> section, links in the stylesheet, starts the <body> section, displays an image for the banner, and then writes the navigation links under the banner image.

Take a look at one of the navigation links:

<span class="nav-item"><a href="/listconferences">Upcoming Conferences</a></span>

If you were to deploy the application now and try it out, the link above would not work, because the URL /listconferences is not defined yet. You will define it soon. For now, continue inspecting the other template files in same directory.

You will notice that base.template has the following tags:

{% block content %}
{% endblock %}

This allows all other templates to extend the base template by simply defining the named block, which you would do as follows:

{% extends "base.template" %}
{% block content %}
. . . page content ...
{% endblock %}

3. Take a look at app.yaml. It defines static file and directory handlers for the css and images directory, handlers for other parts of the application, and the libraries used by the application.

- url: /images
  static_dir: images

- url: /css
  static_dir: css

- url: /favicon.ico
  static_files: favicon.ico
  upload: favicon.ico
etc...

As you work through the exercises in this training, you will fill out the code for each of these pages.

  • developer.template -- a page for developers
  • home.template -- the home page for the application. Currently it just links to other pages.
  • list_conferences.template -- this page will eventually list the conferences that have been scheduled.
  • schedule_conference.template -- this page will eventually display a form for scheduling a new conference.
  • user_profile.template -- this page will eventually display the user’s profile, including the topics that they are interested in learning about.
  • venues.template -- this page will eventually display information about the venues for conferences.

Set the Application ID

Every application must have an “application ID,” which corresponds to one of the IDs that you have created on https://appengine.google.com/. Every developer can have up to ten application IDs when operating in the free tier. During development and testing, it’s a good idea to recycle application IDs whenever possible so that you don’t use up all your free application IDs.

1. Get an Application ID to use for the Conference Central application.

Note: Use a different application ID than you used for the previous exercise.

Since the application you deployed in the previous exercise was in a finished state, it is important to use a different application ID as you build the conference yourself, to avoid having the application be in an inconsistent state.

Go to https://appengine.google.com/ and review your existing application IDs. Either pick an existing one or create a new one to use for the Conference Central application.

2. Update your application to use your Application ID.

Java:

  1. In the war > WEB-INF folder, edit appengine-web.xml.
  2. Update the value of the <application> tag to be your own application ID:
    <application>yourappid</application>

Python:

  1. Edit app.yaml.
  2. Change the value of the application element to be your own application ID:
    application: yourappid

Configure Your Application

Before deploying the application, you need to configure it in web.xml (Java) or app.yaml (Python).

[ Go to Python steps ]

Java: Configure the Servlets and JSPs

You can configure servlets and JSPs in web.xml, to specify which class or file implements each servlet or JSP and to map them to a URL. You can also configure them to control who can access them, but you will do that in the next exercise.

Note: You will not be able to run your application on localhost or deploy it to App Engine until you have fixed the web.xml configuration file.

1. Edit war > WEB-INF > web.xml.
You see a placeholder configuration for a servlet, and another one for a JSP.

<servlet>
  <servlet-name>ServletName</servlet-name>
  <servlet-class>com.google.cloud.training.conference.ServletClass</servlet-class>
</servlet>
<servlet-mapping>
  <servlet-name>ServletName</servlet-name>
  <url-pattern>/servlet_url</url-pattern>
</servlet-mapping>

<servlet>
  <servlet-name>JSPName</servlet-name>
  <jsp-file>jsp_path</jsp-file>
</servlet>

<servlet-mapping>
  <servlet-name>JSPName</servlet-name>
  <url-pattern>/jsp_url</url-pattern>
</servlet-mapping>

2. Edit the existing servlet configuration in web.xml to contain the settings for the SaveUserProfile.java servlet by entering the configuration code shown below. This causes the servlet to be invoked by the /saveprofile URL.

<servlet>
  <servlet-name>SaveUserProfile</servlet-name
<servlet-class>com.google.cloud.training.conference.SaveUserProfile</servlet-class>

</servlet>
<servlet-mapping>

  <servlet-name>SaveUserProfile</servlet-name>
  <url-pattern>/saveprofile</url-pattern>
</servlet-mapping>

Note: be sure to update the existing placeholder servlet mappings for ServletName and JSPName. If you leave them in place the application will not be able to deploy.

3. Add a configuration for homepage.jsp as follows:

<servlet>
  <servlet-name>HomePage</servlet-name>
  <jsp-file>/homepage.jsp</jsp-file>
</servlet>
<servlet-mapping>
  <servlet-name>HomePage</servlet-name>
  <url-pattern>/homepage</url-pattern>
</servlet-mapping>

4. Add <servlet> and <servlet-mapping> tags for the other JSPs that need to be accessible through a URL. You can copy and paste the configuration code for homepage.jsp and replace values as appropriate. Select an appropriate servlet name for each JSP configuration, and map the servlet to the url_pattern as follows:

  • developer.jsp is accessed at /developer
  • listconferences.jsp is accessed at /listconferences
  • scheduleconference.jsp is accessed at /scheduleconference
  • venues.jsp is accessed at /venues
  • userprofile.jsp is accessed at /userprofile

5. Update the <welcome-file> tag to be homepage.jsp:

<welcome-file-list>
  <welcome-file>homepage.jsp</welcome-file>
</welcome-file-list>

app = webapp2.WSGIApplication([

Run the Application

The next step is to run the application on localhost.

1. Run the application on the development server on your localhost machine

For detailed instructions, see Deploying an App Engine Application.

2. Test the navigation links.

Click on each of the navigation links in the header, and also test the links in the home page. At this stage, each of the pages should display but the content is limited. Any forms you see are not active yet.

3. Deploy the application to App Engine.

For detailed instructions, see Deploying an App Engine Application.

4. Again, click through each of the navigation links to check that mapping has been configured properly.

5. In the Admin Console, view the versions page. You should see one version.

Note: In later exercises, as you deploy new versions, remember to check the versions page and makes sure the default version is selected.

Use the Admin Console to Monitor your Application

Now that you have deployed your application in production, this is a good opportunity to start learning your way around the Admin Console.

View Charts

  1. Open the Admin Console for your application by going to https://appengine.google.com/ and selecting your application.
    The Admin Console opens at the Dashboard page.
  2. In the Dashboard page, view the “Requests per second” graph.
    1. Set the time period to 30 minutes.
    2. You might need to run the application and view a few pages to generate some requests.
  3. Click through each of the different types of charts.
  4. Refresh the page, then scroll down to the section “Current Load.” You can see the statistics for each of the pages.

View Logs

  1. In the Admin Console, In the Main section, go to the Logs pages, and review the logs.
    Every request to the application is recorded in the logs.
  2. In the deployed Conference Central application, go to any of the pages.
  3. Back in the logs, confirm your request was recorded.

View quotas and billing settings

  1. In the Admin Console, in the Main section in the left hand navigation panel, select Quota Details. Scroll down the page to see all the things that have a daily quota.
  2. In the Billing (or Resources) section in the left hand navigation panel, select Billing Settings (or Billing Status).
  3. Select the link to set up billing, then click Enable Billing. Review the data you can set when enabling billing.
  4. Change the daily budget, and notice that the weekly charges update.
  5. There is no need to enable billing.

View application settings

  1. In the Admin Console, in the Administration section in the left hand navigation panel, select Application Settings.
  2. Scroll down the page to see the settings that you can change. Notice that you can delete or duplicate your application on this page.
  3. Change your application’s title and save the changes.
  4. Optionally, if you preferred your previous title, change the application title back to what it was.
  5. In the Administration section in the left hand navigation panel, select Permissions.
  6. Optionally, invite another developer to collaborate on the application with you. Notice the different roles that a developer can have. (In the next exercise, you will limit access to the developers page of your application to developers who have a role in developing your application).
  7. In the Administration section in the left hand navigation panel, select Admin Logs to see a history of things that administrators have done with the application.
  8. Select the Events dropdown menu to see the different kinds of events that can be listed.

Conclusion

In this exercise, you got started building the Conference Central application, and you practiced deploying it and using the Admin Console to view settings and usage.

In the next exercise, you will set up access control for some of the pages to restrict access to logged in users for some of the pages.
